Fletcher

Compact, positional row buffers for Apache Arrow — generated from Protocol Buffers, with schema transport, pub/sub delivery, and columnar batching built in.

C++20 TypeScript Apache Arrow Protocol Buffers Schema Transport WebSocket Pub/Sub

Overview

Fletcher bridges two worlds: Protocol Buffers for message definition and Apache Arrow for in-memory analytics. A protoc compiler plugin reads your .proto files and generates C++ wrapper classes that serialize structured data into compact, positional binary row buffers backed by Arrow schemas.

These row buffers are designed for scenarios where data arrives one row at a time — telemetry streams, event buses, sensor networks — but is consumed in columnar batches for analytics, storage, or forwarding.

The system provides six layers that compose cleanly:

  1. Codec — encodes Arrow scalars into a compact positional binary row format and decodes them back. The publisher’s schema is delivered to subscribers automatically, guaranteeing matching schemas on both sides.
  2. Protoc Plugin — generates typed C++ message classes, TypeScript interfaces, publisher/subscriber classes, and Arrow schemas from .proto definitions.
  3. PubSub & Driver — an abstract provider interface (with a Fast DDS implementation) for publishing and subscribing, with provider-level schema transport, multi-subscriber fan-out, and subscription IDs.
  4. WebGateway — a Boost.Beast WebSocket server that exposes the Driver to browser clients over a split text/binary protocol.
  5. WebClient — a TypeScript client library with a pure-TypeScript positional codec and two interchangeable backends (plain objects or Apache Arrow JS).
  6. Batcher — accumulates individual rows into Arrow Tables via a write-ahead log, flushing in configurable batches.

Quick Start

1. Define your message

// telemetry.proto
syntax = "proto3";
package myapp;

import "google/protobuf/empty.proto";

message SensorReading {
    int32  sensor_id    = 1;
    double temperature  = 2;
    string location     = 3;
    optional double humidity = 4;  // nullable
}

service SensorFeed {
    rpc Stream(stream SensorReading) returns (google.protobuf.Empty);
}

2. Generate code

The protoc-gen-fletcher plugin runs during your CMake build. For each .proto file it produces a .fletcher.pb.h header containing:

  • An Arrow schema function (SensorReadingArrowRowSchema())
  • A typed row class (SensorReadingArrowRow) with chainable setters, Encode(), and a decoding constructor
  • Publisher/Subscriber classes for each service RPC

3. Encode and decode

#include "telemetry.fletcher.pb.h"

// Build a row with typed setters
myapp::SensorReadingArrowRow row;
row.set_sensor_id(42)
   .set_temperature(23.5)
   .set_location("Room 101")
   .set_humidity(55.3);

// Encode to a compact binary buffer
fletcher::EncodedRow encoded = row.Encode();

// Decode back
myapp::SensorReadingArrowRow restored(encoded);

4. Publish and subscribe

#include <fast_dds_pubsub_provider.hpp>

auto provider = std::make_shared<fletcher::FastDDSPubSubProvider>();

// Publisher
myapp::SensorFeed_StreamPublisher pub(provider);
pub.Publish(row);

// Subscriber
myapp::SensorFeed_StreamSubscriber sub(provider);
sub.Subscribe([](const fletcher::Envelope& env) {
    myapp::SensorReadingArrowRow msg(env.row);
    std::cout << "Temperature: " << msg.temperature() << "\n";
});

5. Batch into Arrow Tables

fletcher::SQLiteWAL wal(":memory:");
fletcher::GenericRowBatcher batcher(
    myapp::SensorReadingArrowRowSchema(),
    wal,
    /*batch_size=*/1000,
    [](std::shared_ptr<arrow::Table> table) -> bool {
        // Write to Parquet, send to a data lake, etc.
        return true;
    });

sub.Subscribe([&](const fletcher::Envelope& env) {
    batcher.Append(env.row);  // triggers flush every 1000 rows
});

(columnar) | | | v | v WebClient (TS) +---- Decode() <-- Subscriber positional codec + (schema received Object / Arrow JS from publisher)

Each layer is independent. You can use the codec without pub/sub, the batcher without the protoc plugin, the web gateway without DDS, or the full pipeline end-to-end.

Wire Format (Positional)

Every EncodedRow is a compact, positional binary buffer (little-endian throughout). Because the publisher’s schema is delivered to subscribers via schema transport, both sides are guaranteed the same schema. This allows the wire format to omit field numbers, wire type tags, and schema hashes — fields are written in schema order with a compact null bitfield.

[NULL_BITFIELD : ceil(num_fields / 8) bytes, LSB-first bit order]

For each field in schema order (skip if null bit set):
  [PAYLOAD bytes]   — size determined by field type from schema

Payload encoding by kind

KindEncoding
Fixed-size scalars (int, float, bool, ...)Raw little-endian bytes (size known from schema type)
String / Binary[LEN:4][DATA:LEN]
Struct[NULL_BITFIELD:ceil(n/8)] + per-field payloads (recursive positional)
List / Large List[COUNT:4][NULL_BITFIELD:ceil(count/8)] + per-element payloads
Fixed-size List[NULL_BITFIELD:ceil(fixed_size/8)] + per-element payloads
Map[COUNT:4] + key payloads + [NULL_BITFIELD:ceil(count/8)] + value payloads
Union[TYPE_CODE:1] + active child payload

Wire size savings

For a typical 10-field row (6 fixed scalars, 2 strings, 1 struct, 1 list):

Schema Transport

The positional wire format requires both publisher and subscriber to share the same schema. Fletcher solves this via provider-level schema transport: when a publisher creates a topic, the Arrow schema is delivered alongside the topic. When a subscriber joins, the provider returns the schema automatically.

How it works

  1. Publisher calls CreateTopic(segments, schema) — the provider stores the schema alongside the data topic.
  2. Subscriber calls Subscribe(segments, callback) — the provider returns a SubscriptionResult containing the publisher’s schema. No need for the subscriber to know the schema in advance.
  3. Both sides use the same schema for the PositionalCodec, so the compact positional wire format works without any per-field metadata.

FastDDS implementation

The FastDDS provider uses a companion DDS topic for schema delivery. For each data topic foo/bar, a companion topic foo/bar/__schema is created with TRANSIENT_LOCAL durability and KEEP_LAST(1) QoS. The serialized Arrow IPC schema is published once and retained for late-joining subscribers.

WebGateway

When a WebSocket client subscribes, the subscribed response includes the schema in two forms:

  • schema — a JSON object with field names, wire types, nullability, and composite structure (for easy consumption in JS)
  • schemaIpc — base64-encoded Arrow IPC bytes (for full fidelity when needed)
{
  "type": "subscribed",
  "subId": "1",
  "topic": "myapp/SensorFeed/Stream",
  "schemaIpc": "QUFBQUFBP...",
  "schema": {
    "fields": [
      {"name": "sensor_id", "fieldNumber": 1, "wireType": 4, "nullable": false},
      ...
    ]
  }
}
Legacy tagged codec: The older self-describing tagged wire format (RowCodec) and its schema evolution logic are still present in the codebase for backward compatibility but are no longer used by generated code.

Attachments (Sidecar Blobs)

Large binary data — images, audio, firmware blobs — should not be embedded in the Arrow row structure. Instead, the message references a blob by key (a plain string field), while the actual data travels alongside it as an attachment.

// The Arrow row contains just a reference
row.set_image_ref("frame_001");

// The blob travels as a sidecar attachment
auto image = std::make_shared<const std::vector<uint8_t>>(load_image());
pub.Publish(row, {{"frame_001", image}});

On the subscriber side, the Envelope contains both the encoded row and the attachments map:

sub.Subscribe([&](const fletcher::Envelope& env) {
    MyMessage msg(env.row);                              // structured data
    auto it = env.attachments.find(msg.image_ref());
    if (it != env.attachments.end())
        save_image(*it->second);                       // blob data

    batcher.Append(env.row);                             // batcher sees only the row
});

Types

TypeDefinitionPurpose
Blobshared_ptr<const vector<uint8_t>>Zero-copy binary data
Attachmentsunordered_map<string, Blob>Key/value blob pairs

SerializeEnvelope() and DeserializeEnvelope() provide a wire format for transport providers that send everything as a single byte stream. The codec and batcher are completely unaware of attachments — they remain a transport-layer concern.

Protoc Plugin

The protoc-gen-fletcher plugin is a standard protobuf compiler plugin. During a CMake build, protoc invokes it for each .proto file and it emits a .fletcher.pb.h C++ header. With the ts option, it also emits a .fletcher.ts TypeScript file.

What gets generated (C++)

For each message:

  • A free function {Message}ArrowRowSchema() returning the Arrow schema with field_number metadata on each field and proto_package/proto_message on the schema.
  • A class {Message}ArrowRow with:
    • Chainable setters (set_*(), clear_*() for optionals)
    • Encode() → EncodedRow
    • Constructor from EncodedRow (decodes)

For each eligible service RPC (client-streaming, returns google.protobuf.Empty):

  • {Service}_{Method}Publisher — wraps PubSubProvider::Publish, provides Publish(row) and Publish(row, attachments)
  • {Service}_{Method}Subscriber — wraps PubSubProvider::Subscribe, delivers Envelope to callback

Unsupported constructs

  • oneof — use separate optional fields instead
  • Recursive messages — detected at generation time, skipped with a comment
  • google.protobuf.Any and google.protobuf.Struct — dynamically typed

TypeScript generation (--fletcher_opt=ts)

When the ts option is passed, the plugin also emits a .fletcher.ts file alongside the C++ header:

PositionalCodec

The core encode/decode engine. You rarely use it directly — the generated classes call it internally — but it is available for manual use with hand-crafted Arrow schemas.

auto schema = arrow::schema({
    arrow::field("id",   arrow::int32(),   false),
    arrow::field("name", arrow::utf8(),    true),
});

fletcher::PositionalCodec codec(schema);

// Encode
fletcher::EncodedRow row = codec.EncodeRow({
    std::make_shared<arrow::Int32Scalar>(42),
    std::make_shared<arrow::StringScalar>("Alice"),
});

// Decode
fletcher::ArrowRow scalars = codec.DecodeRow(row);
// scalars[0] is an Int32Scalar, scalars[1] is a StringScalar
Positional decoding: Because both encoder and decoder share the same schema (delivered via schema transport), the codec reads fields by position using the null bitfield — no field matching, no type tags, no per-field overhead.

Pub/Sub & Driver

The PubSubProvider is an abstract interface that decouples message transport from encoding. Implement it to plug in any message bus.

struct SubscriptionResult {
    std::shared_ptr<arrow::Schema> schema;  // publisher's schema
};

class PubSubProvider {
public:
    virtual void CreateTopic(
        const std::vector<std::string>& topic_segments,
        std::shared_ptr<arrow::Schema> schema) = 0;

    virtual void Publish(
        const std::vector<std::string>& topic_segments,
        const Envelope& envelope) = 0;

    using SubscribeCallback = std::function<void(const Envelope&)>;

    virtual SubscriptionResult Subscribe(
        const std::vector<std::string>& topic_segments,
        SubscribeCallback callback) = 0;

    virtual void Unsubscribe(
        const std::vector<std::string>& topic_segments) = 0;
};

Topic names are a list of string segments so each provider can join them with its preferred separator (e.g., / for DDS, . for MQTT). The generated code uses {package, service, method} as segments.

Driver

The Driver class wraps a PubSubProvider with higher-level features needed by the WebGateway and application code:

  • Multi-subscriber fan-out — multiple subscribers per topic, each receiving its own callback. The provider sees only one subscription per topic.
  • Subscription IDs + schema — each Subscribe() call returns a SubscribeResult{subscription_id, schema} with the publisher’s Arrow schema and a unique ID for targeted unsubscribe.
  • Auto-registration — subscribing to an unknown topic auto-registers it with the schema received from the provider.
  • Topic registry — ListTopics() and HasTopic() for discovery.
  • Lifecycle management — automatically unsubscribes from the provider when the last subscriber for a topic is removed.
auto provider = std::make_shared<fletcher::FastDDSPubSubProvider>();
auto driver   = std::make_shared<fletcher::Driver>(provider);

driver->CreateTopic({"myapp", "sensors"}, schema);

auto [sub1, schema1] = driver->Subscribe({"myapp", "sensors"}, callback1);
auto [sub2, schema2] = driver->Subscribe({"myapp", "sensors"}, callback2);

driver->Unsubscribe(sub1);  // sub2 still active

WebGateway

A Boost.Beast WebSocket server that exposes the Driver to browser clients over a split text/binary WebSocket protocol. Control messages use JSON text frames for simplicity and debuggability; data messages (publish and message delivery) use binary frames to avoid base64 overhead.

fletcher::WebGatewayOptions opts;
opts.port = 9090;
opts.io_threads = 2;

fletcher::WebGateway gw(driver, opts);
gw.Start();   // non-blocking, spawns IO threads
// ... gateway serves until:
gw.Stop();

Sub IDs are stringified in JSON to avoid JavaScript Number precision loss. All multi-byte integers in binary frames are little-endian. Each WebSocket connection gets its own session with independent subscriptions. Subscriptions are automatically cleaned up when the connection closes.

WebClient (TypeScript)

The @fletcher/web-client npm package connects to the WebGateway and decodes/encodes the Fletcher positional wire format in the browser. It uses a pure-TypeScript positional codec and offers two interchangeable decoder backends:

  • Object backend — zero dependencies, decodes rows into plain Record<string, unknown> objects.
  • Arrow backend — (planned) decodes into Apache Arrow JS RecordBatch via the optional apache-arrow peer dependency.

Quick example

import { FletcherClient } from '@fletcher/web-client';
import { ISensorReading,
         SensorFeed_StreamTopic } from './telemetry.fletcher.js';

const client = new FletcherClient({ url: 'ws://localhost:9090' });
await client.connect();

// Schema is received automatically from the server on subscribe
const subId = await client.subscribe<ISensorReading>(
  SensorFeed_StreamTopic,
  (msg, attachments) => {
    console.log(msg.sensor_id, msg.temperature);
  }
);

// Later...
await client.unsubscribe(subId);
client.close();

The subscribe method also accepts an explicit SchemaDescriptor as the second argument if you want to override the server-provided schema.

Batcher & Write-Ahead Log

The batcher accumulates individual EncodedRow buffers and flushes them as columnar arrow::Table batches. A write-ahead log (WAL) backed by SQLite provides durability between flushes.

fletcher::SQLiteWAL wal("wal.db");  // or ":memory:" for in-memory
fletcher::GenericRowBatcher batcher(
    schema,
    wal,
    /*batch_size=*/1000,
    [](std::shared_ptr<arrow::Table> table) -> bool {
        write_parquet(table);
        return true;  // return false to signal failure
    });

// Thread-safe: call from subscriber callbacks or any thread
batcher.Append(encoded_row);

Flush behavior

  • Append() is thread-safe (mutex-protected)
  • When row_count reaches batch_size, the flush callback is dispatched on a background thread
  • The callback returns bool: true = success, false = failure
  • The destructor joins all pending flush futures
WAL column types: The SQLite WAL maps Arrow types to SQLite types (INTEGER, REAL, TEXT, BLOB). Composite types (struct, list, map) are not supported in the WAL and will throw at log creation time.

FastDDS Provider

A concrete PubSubProvider implementation using eProsima Fast DDS for reliable, real-time publish/subscribe over RTPS.

auto provider = std::make_shared<fletcher::FastDDSPubSubProvider>(
    /*domain_id=*/0,
    /*max_payload_bytes=*/1024 * 1024  // 1 MB
);

QoS configuration

PolicySettingEffect
ReliabilityRELIABLEMiddleware retransmits lost samples
HistoryKEEP_ALLAll samples retained until acknowledged
DurabilityTRANSIENT_LOCALLate-joining subscribers receive buffered history

The provider serializes each Envelope (row + attachments) into a single DDS sample using the envelope wire format. On the receiver side, it deserializes back into an Envelope before invoking the subscriber callback.

Schema delivery

For each data topic, the provider creates a companion DDS topic with the suffix /__schema using TRANSIENT_LOCAL durability and KEEP_LAST(1) QoS. The publisher serializes the Arrow schema as an IPC stream and publishes it to the companion topic. When a subscriber joins, the provider reads the retained schema sample before creating the data reader, and returns it in the SubscriptionResult.

C API & Rust Bindings

The codec and batcher expose a pure-C API for cross-language interop. Schemas and record batches are exchanged as Arrow IPC stream bytes.

// C API (arrow_row_codec_capi.h)
ArrowRowCodec* codec = arrow_row_codec_new(schema_ipc, schema_len, &err);
arrow_row_codec_encode_row(codec, row_ipc, row_len, &out, &out_len, &err);
arrow_row_codec_decode_row(codec, buf, buf_len, &out_ipc, &out_len, &err);
arrow_row_codec_free(codec);

Rust bindings wrap the C API with safe types via the arrow-row-codec and arrow-row-batcher crates (under Codec/bindings/rust/ and Batcher/bindings/rust/). They use the cmake crate to link against the C++ libraries at build time.

Building

Prerequisites

  • CMake 3.15+
  • Conan 2 package manager
  • A C++20 compiler (MSVC 2022, GCC 12+, Clang 15+)

Steps

# 1. Install dependencies via Conan
conan install . --build=missing --output-folder=build

# 2. Configure with CMake (uses the Conan-generated toolchain)
cmake --preset conan-default

# 3. Build
cmake --build build --config Release

WebClient (TypeScript)

# Install dependencies and run tests
cd WebClient
npm install
npm test            # 37 Vitest tests

# TypeScript compilation
npm run build:ts

# Generate TypeScript from .proto files
protoc --plugin=protoc-gen-fletcher=build/ProtoPlugin/Release/protoc-gen-fletcher \
       --fletcher_out=. --fletcher_opt=ts \
       -I proto your_messages.proto
Windows / Visual Studio: The root CMakeLists.txt forces /MD (MultiThreadedDLL) for all targets and patches Conan's imported targets so that Release-only packages (protobuf, Fast DDS) work correctly in Debug builds. Multi-config generators are fully supported.
