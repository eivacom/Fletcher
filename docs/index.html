Fletcher

Compact, positional row buffers for Apache Arrow — generated from Protocol Buffers, with nanoarrow-based schema transport, pub/sub delivery, and columnar batching built in.

C++20 TypeScript Nanoarrow Apache Arrow Protocol Buffers Schema Transport WebSocket Edge-Ready Pub/Sub

Overview

Fletcher bridges two worlds: Protocol Buffers for message definition and Apache Arrow for in-memory analytics. A protoc compiler plugin reads your .proto files and generates C++ wrapper classes that serialize structured data into compact, positional binary row buffers backed by Arrow schemas.

These row buffers are designed for scenarios where data arrives one row at a time — telemetry streams, event buses, sensor networks — but is consumed in columnar batches for analytics, storage, or forwarding.

The core pub/sub, codec, generated code, WebGateway, and transport providers depend only on nanoarrow (~100 KB), making them suitable for edge and embedded deployment. Server-side features (batching, view classes) use the full Apache Arrow C++ library through a separate PubSubArrow adapter.

The system provides seven layers that compose cleanly:

  1. Core — a header-only library with PositionalWriter/PositionalReader for direct serialization, WriteBuffer, Envelope, and EncodedRow type aliases. No dependencies.
  2. ArrowBridge — the Codec class for server-side ArrowRow encode/decode (Arrow C++), plus ArrowRowView helpers and CRS utilities.
  3. Protoc Plugin — generates typed C++ message classes (.fletcher.pb.h, nanoarrow only) and optional Arrow C++ view classes (.fletcher.arrow.pb.h), plus TypeScript interfaces.
  4. PubSub Provider — an abstract transport interface operating on raw bytes and nanoarrow OwnedSchema, with schema transport, zero-copy RowEncoder publishing, and raw-bytes subscriber callbacks.
  5. PubSubArrow — a server-side wrapper that adds Arrow C++ convenience (arrow::Schema, ArrowRow encode/decode) on top of the nanoarrow provider.
  6. WebGateway — a Boost.Beast WebSocket server that exposes the Driver to browser clients over a split text/binary protocol.
  7. WebClient — a TypeScript client library with a pure-TypeScript positional codec and two interchangeable backends (plain objects or Apache Arrow JS).
  8. Batcher — accumulates individual rows into Arrow Tables via a write-ahead log, flushing in configurable batches.

Quick Start

1. Define your message

// telemetry.proto
syntax = "proto3";
package myapp;

import "google/protobuf/empty.proto";

message SensorReading {
    int32  sensor_id    = 1;
    double temperature  = 2;
    string location     = 3;
    optional double humidity = 4;  // nullable
}

service SensorFeed {
    rpc Stream(stream SensorReading) returns (google.protobuf.Empty);
}

2. Generate code

The protoc-gen-fletcher plugin runs during your CMake build. For each .proto file it produces two headers:

  • .fletcher.pb.h — edge-compatible message class, schema function, publisher, subscriber. Depends only on nanoarrow.
  • .fletcher.arrow.pb.h — server-side view class wrapping ArrowRow with typed getters for RecordBatch, Table, and scalar vectors. Depends on Arrow C++.

3. Encode and decode

#include "telemetry.fletcher.pb.h"

// Build a row with typed setters
fletcher_gen::myapp::SensorReading row;
row.set_sensor_id(42)
   .set_temperature(23.5)
   .set_location("Room 101")
   .set_humidity(55.3);

// Encode to a compact binary buffer (no Arrow C++ needed)
fletcher::EncodedRow encoded = row.Encode();

// Decode back from raw bytes
fletcher_gen::myapp::SensorReading restored(encoded);

4. Publish and subscribe

#include <fast_dds_pubsub_provider.hpp>

auto provider = std::make_shared<fletcher::FastDDSPubSubProvider>();  // implements PubSub

// Publisher (creates topic with nanoarrow schema, zero-copy encode)
fletcher_gen::myapp::SensorFeed_StreamPublisher pub(provider);
pub.Publish(row);

// Subscriber (receives typed message, decoded from raw bytes)
fletcher_gen::myapp::SensorFeed_StreamSubscriber sub(provider);
sub.Subscribe([](fletcher_gen::myapp::SensorReading msg, fletcher::Attachments att) {
    std::cout << "Temperature: " << msg.temperature() << "\n";
});

5. Batch into Arrow Tables

fletcher::SQLiteWAL wal(":memory:");
fletcher::GenericRowBatcher batcher(
    fletcher_gen::myapp::SensorReadingSchema(),
    wal,
    /*batch_size=*/1000,
    [](std::shared_ptr<arrow::Table> table) -> bool {
        // Write to Parquet, send to a data lake, etc.
        return true;
    });

sub.Subscribe([&](fletcher_gen::myapp::SensorReading msg, fletcher::Attachments att) {
    batcher.Append(msg.Encode());  // triggers flush every 1000 rows
});

RowBatcher Browser | | (accumulation) | | v | v +---- Decode() <-- arrow::Table WebClient (TS) (PositionalReader (columnar) positional codec + or Codec) Object / Arrow JS

Each layer is independent. You can use the codec without pub/sub, the batcher without the protoc plugin, the web gateway without DDS, or the full pipeline end-to-end. The core path (codec, pub/sub, generated message code, DDS provider) requires only nanoarrow, enabling deployment on edge and embedded devices.

Nanoarrow & Edge Deployment

The full Apache Arrow C++ library is ~100 MB — impractical for edge devices, microcontrollers, or space-constrained embedded systems. Nanoarrow is a lightweight (~100 KB) C library that implements the Arrow type system, schema construction, and IPC format without the full C++ runtime.

Fletcher's core pub/sub path — generated message code, the PubSub interface, schema IPC serialization, and transport providers like FastDDS — depends only on nanoarrow. This means an edge binary can publish and subscribe to Arrow-typed data streams without linking the full Arrow C++ library.

Two-tier architecture

OwnedSchema

Schemas are represented as fletcher::OwnedSchema — an RAII wrapper around nanoarrow's C ArrowSchema struct. It is zero-copy convertible to/from Arrow C++'s shared_ptr<arrow::Schema> via the Arrow C Data Interface (arrow::ImportSchema / arrow::ExportSchema).

Wire Format (Positional)

Every EncodedRow is a compact, positional binary buffer (little-endian throughout). Because the publisher’s schema is delivered to subscribers via schema transport, both sides are guaranteed the same schema. This allows the wire format to omit field numbers, wire type tags, and schema hashes — fields are written in schema order with a compact null bitfield.

[NULL_BITFIELD : ceil(num_fields / 8) bytes, LSB-first bit order]

For each field in schema order (skip if null bit set):
  [PAYLOAD bytes]   — size determined by field type from schema

Payload encoding by kind

KindEncoding
Fixed-size scalars (int, float, bool, ...)Raw little-endian bytes (size known from schema type)
String / Binary[LEN:4][DATA:LEN]
Struct[NULL_BITFIELD:ceil(n/8)] + per-field payloads (recursive positional)
List / Large List[COUNT:4][NULL_BITFIELD:ceil(count/8)] + per-element payloads
Fixed-size List[NULL_BITFIELD:ceil(fixed_size/8)] + per-element payloads
Map[COUNT:4] + key payloads + [NULL_BITFIELD:ceil(count/8)] + value payloads
Union[TYPE_CODE:1] + active child payload

Wire size savings

For a typical 10-field row (6 fixed scalars, 2 strings, 1 struct, 1 list):

Schema Transport

The positional wire format requires both publisher and subscriber to share the same schema. Fletcher solves this via provider-level schema transport: when a publisher creates a topic, the nanoarrow OwnedSchema is delivered alongside the topic. When a subscriber joins, the provider returns the schema automatically.

How it works

  1. Publisher calls CreateTopic(segments, OwnedSchema) — the provider stores the schema alongside the data topic.
  2. Subscriber calls Subscribe(segments, callback) — the provider returns a SubscriptionResult containing the publisher’s schema as an OwnedSchema. No need for the subscriber to know the schema in advance.
  3. Both sides use the same schema for the positional wire format, so the compact encoding works without any per-field metadata.

FastDDS implementation

The FastDDS provider uses a companion DDS topic for schema delivery. For each data topic foo/bar, a companion topic foo/bar/__schema is created with TRANSIENT_LOCAL durability and KEEP_LAST(1) QoS. The schema is serialized as Arrow IPC bytes via nanoarrow and published once, retained for late-joining subscribers.

WebGateway

When a WebSocket client subscribes, the subscribed response includes the schema in two forms:

  • schema — a JSON object with field names, wire types, nullability, and composite structure (for easy consumption in JS)
  • schemaIpc — base64-encoded Arrow IPC bytes (for full fidelity when needed)
{
  "type": "subscribed",
  "subId": "1",
  "topic": "myapp/SensorFeed/Stream",
  "schemaIpc": "QUFBQUFBP...",
  "schema": {
    "fields": [
      {"name": "sensor_id", "fieldNumber": 1, "wireType": 4, "nullable": false},
      ...
    ]
  }
}
Legacy tagged codec: The older self-describing tagged wire format (RowCodec) and its schema evolution logic have been removed from the codebase — fully superseded by the positional codec and generated code.

Attachments (Sidecar Blobs)

Large binary data — images, audio, firmware blobs — should not be embedded in the Arrow row structure. Instead, the message references a blob by key (a plain string field), while the actual data travels alongside it as an attachment.

// The Arrow row contains just a reference
row.set_image_ref("frame_001");

// The blob travels as a sidecar attachment
auto image = std::make_shared<const std::vector<uint8_t>>(load_image());
pub.Publish(row, {{"frame_001", image}});

On the subscriber side, attachments arrive alongside the decoded message:

sub.Subscribe([&](MyMessage msg, fletcher::Attachments att) {
    auto it = att.find(msg.image_ref());
    if (it != att.end())
        save_image(*it->second);                       // blob data
});

Types

TypeDefinitionPurpose
Blobshared_ptr<const vector<uint8_t>>Zero-copy binary data
Attachmentsunordered_map<string, Blob>Key/value blob pairs

SerializeEnvelope() and DeserializeEnvelope() provide a wire format for transport providers that send everything as a single byte stream. The codec and batcher are completely unaware of attachments — they remain a transport-layer concern.

Protoc Plugin

The protoc-gen-fletcher plugin is a standard protobuf compiler plugin. During a CMake build, protoc invokes it for each .proto file and it emits two C++ headers. With the ts option, it also emits a .fletcher.ts TypeScript file.

Generated files

FileDepends onContents
.fletcher.pb.h nanoarrow only Schema function (OwnedSchema), typed row class with Encode()/EncodeTo(), decoding constructor, publisher/subscriber classes
.fletcher.arrow.pb.h Arrow C++ Immutable view class with typed getters, constructors from ArrowRow, RecordBatch, and Table
.fletcher.ts — TypeScript interface, SchemaDescriptor, topic constant

What gets generated (C++ — .fletcher.pb.h)

For each message:

  • A free function {Message}Schema() returning a fletcher::OwnedSchema (nanoarrow ArrowSchema) with field_number metadata on each field and proto_package/proto_message on the schema.
  • A class {Message} (in the fletcher_gen namespace) with:
    • Chainable setters (set_*(), clear_*() for optionals)
    • Encode() → EncodedRow (via PositionalWriter)
    • EncodeTo(WriteBuffer&) for zero-copy encoding into a caller-provided buffer
    • Constructor from EncodedRow (via PositionalReader)
    • Constructor from raw (const uint8_t*, size_t)

For each eligible service RPC (client-streaming, returns google.protobuf.Empty):

  • {Service}_{Method}Publisher — creates topic with OwnedSchema, publishes via RowEncoder zero-copy path
  • {Service}_{Method}Subscriber — subscribes to raw bytes, delivers typed {Message} + Attachments to callback

What gets generated (C++ — .fletcher.arrow.pb.h)

For each message:

  • A class {Message}View with:
    • Constructor from ArrowRow (vector of Arrow scalars)
    • Constructor from shared_ptr<arrow::Scalar> (struct scalar)
    • Constructor from arrow::RecordBatch + row index
    • Constructor from arrow::Table + row index
    • Typed zero-copy getters (returns string_view for strings, nested view classes for structs, ArrowRowViewList for lists)

Unsupported constructs

  • oneof — use separate optional fields instead
  • Recursive messages — detected at generation time, skipped with a comment
  • google.protobuf.Any and google.protobuf.Struct — dynamically typed

Codec & Positional I/O

There are two levels of positional encode/decode, for different deployment targets:

PositionalWriter / PositionalReader (nanoarrow, edge-compatible)

Header-only utilities in core/positional_io.hpp. Used by generated EncodeTo() and decoding constructors. No Arrow C++ dependency.

// Zero-copy encoding into a caller-provided buffer
fletcher::VectorWriteBuffer buf(encoded_row);
fletcher::PositionalWriter w(buf, /*num_fields=*/4);
w.WriteInt32(42);
w.WriteDouble(23.5);
w.WriteString("Room 101");
w.SetNull(3);
w.Finish();

// Decoding
fletcher::PositionalReader r(encoded_row.data(), encoded_row.size(), 4);
int32_t id   = r.ReadInt32();
double  temp = r.ReadDouble();
auto    loc  = r.ReadString();
bool    has3 = !r.IsNull(3);

Codec (Arrow C++, server-side)

The Codec class (in ArrowBridge/) works with ArrowRow (vector<shared_ptr<arrow::Scalar>>) and requires an Arrow C++ schema. Used by PubSubArrow and the view classes for decoding raw bytes into Arrow scalars.

auto schema = arrow::schema({
    arrow::field("id",   arrow::int32(),   false),
    arrow::field("name", arrow::utf8(),    true),
});

fletcher::Codec codec(schema);

// Encode from Arrow scalars
fletcher::EncodedRow row = codec.EncodeRow({
    std::make_shared<arrow::Int32Scalar>(42),
    std::make_shared<arrow::StringScalar>("Alice"),
});

// Decode to Arrow scalars
fletcher::ArrowRow scalars = codec.DecodeRow(row);
Wire-compatible: Both levels produce byte-identical output. A row encoded via PositionalWriter on an edge device can be decoded by Codec on a server, and vice versa.

PubSub Interface

The PubSub class is an abstract interface that decouples message transport from encoding. It operates on raw bytes and nanoarrow schemas — no Arrow C++ dependency. Implement it to plug in any message bus.

struct SubscriptionResult {
    fletcher::OwnedSchema schema;  // nanoarrow ArrowSchema (RAII)
};

class PubSub {
public:
    virtual void CreateTopic(
        const std::vector<std::string>& topic_segments,
        OwnedSchema schema) = 0;

    // Zero-copy publish: encoder writes directly into provider's buffer
    using RowEncoder = std::function<void(WriteBuffer&)>;

    virtual void Publish(
        const std::vector<std::string>& topic_segments,
        RowEncoder encoder,
        const Attachments& attachments = {}) = 0;

    // Subscriber receives raw bytes, shared schema, and attachments
    using SubscribeCallback = std::function<void(
        const uint8_t* data, size_t len,
        SharedSchema schema,
        Attachments attachments)>;

    virtual SubscriptionResult Subscribe(
        const std::vector<std::string>& topic_segments,
        SubscribeCallback callback) = 0;

    virtual void Unsubscribe(
        const std::vector<std::string>& topic_segments) = 0;
};

Topic names are a list of string segments so each provider can join them with its preferred separator (e.g., / for DDS, . for MQTT). The generated code uses {package, service, method} as segments.

RowEncoder: zero-copy publish

Instead of passing a pre-encoded buffer, the provider calls the RowEncoder lambda with a WriteBuffer&. This allows the provider to supply its own buffer (e.g., a FixedWriteBuffer pointing directly at DDS shared memory), eliminating intermediate copies.

PubSubArrow (Server-Side Adapter)

A thin adapter library that wraps the nanoarrow-based PubSub interface with Arrow C++ convenience types for server-side code. Schemas are converted between OwnedSchema and shared_ptr<arrow::Schema> via the Arrow C Data Interface.

auto provider = std::make_shared<fletcher::FastDDSPubSubProvider>();
fletcher::PubSubArrow arrow_ps(provider);

// Create topic with Arrow C++ schema
arrow_ps.CreateTopic({"myapp", "sensors"}, arrow_schema);

// Publish from ArrowRow
arrow_ps.Publish({"myapp", "sensors"}, arrow_row);

// Subscribe — delivers ArrowRow to callback
auto result = arrow_ps.Subscribe({"myapp", "sensors"},
    [](fletcher::ArrowRow row, fletcher::Attachments att) {
        // row is vector<shared_ptr<arrow::Scalar>>
    });

Features

  • Multi-subscriber fan-out — multiple subscribers per topic, each receiving its own callback. The provider sees only one subscription per topic.
  • Subscription IDs + schema — each Subscribe() call returns a SubscribeResult{subscription_id, arrow::Schema} with the publisher’s Arrow schema and a unique ID for targeted unsubscribe.
  • Auto-registration — subscribing to an unknown topic auto-registers it with the schema received from the provider.
  • Topic registry — ListTopics() and HasTopic() for discovery.
  • Lifecycle management — automatically unsubscribes from the provider when the last subscriber for a topic is removed.
  • ArrowRow encode/decode — uses Codec internally to convert between Arrow scalars and raw bytes.

WebGateway

A Boost.Beast WebSocket server that exposes the Driver to browser clients over a split text/binary WebSocket protocol. The WebGateway depends only on nanoarrow (not Arrow C++) — it walks the nanoarrow ArrowSchema directly to produce the JSON schema response, and passes row data through as raw bytes.

Control messages use JSON text frames for simplicity and debuggability; data messages (publish and message delivery) use binary frames to avoid base64 overhead.

fletcher::WebGatewayOptions opts;
opts.port = 9090;
opts.io_threads = 2;

fletcher::WebGateway gw(driver, opts);
gw.Start();   // non-blocking, spawns IO threads
// ... gateway serves until:
gw.Stop();

Sub IDs are stringified in JSON to avoid JavaScript Number precision loss. All multi-byte integers in binary frames are little-endian. Each WebSocket connection gets its own session with independent subscriptions. Subscriptions are automatically cleaned up when the connection closes.

WebClient (TypeScript)

The @fletcher/web-client npm package connects to the WebGateway and decodes/encodes the Fletcher positional wire format in the browser. It uses a pure-TypeScript positional codec and offers two interchangeable decoder backends:

  • Object backend — zero dependencies, decodes rows into plain Record<string, unknown> objects.
  • Arrow backend — (planned) decodes into Apache Arrow JS RecordBatch via the optional apache-arrow peer dependency.

Quick example

import { FletcherClient } from '@fletcher/web-client';
import { ISensorReading,
         SensorFeed_StreamTopic } from './telemetry.fletcher.js';

const client = new FletcherClient({ url: 'ws://localhost:9090' });
await client.connect();

// Schema is received automatically from the server on subscribe
const subId = await client.subscribe<ISensorReading>(
  SensorFeed_StreamTopic,
  (msg, attachments) => {
    console.log(msg.sensor_id, msg.temperature);
  }
);

// Later...
await client.unsubscribe(subId);
client.close();

The subscribe method also accepts an explicit SchemaDescriptor as the second argument if you want to override the server-provided schema.

Batcher & Write-Ahead Log

The batcher accumulates individual EncodedRow buffers and flushes them as columnar arrow::Table batches. A write-ahead log (WAL) backed by SQLite provides durability between flushes.

fletcher::SQLiteWAL wal("wal.db");  // or ":memory:" for in-memory
fletcher::GenericRowBatcher batcher(
    schema,
    wal,
    /*batch_size=*/1000,
    [](std::shared_ptr<arrow::Table> table) -> bool {
        write_parquet(table);
        return true;  // return false to signal failure
    });

// Thread-safe: call from subscriber callbacks or any thread
batcher.Append(encoded_row);

Flush behavior

  • Append() is thread-safe (mutex-protected)
  • When row_count reaches batch_size, the flush callback is dispatched on a background thread
  • The callback returns bool: true = success, false = failure
  • The destructor joins all pending flush futures
WAL column types: The SQLite WAL maps Arrow types to SQLite types (INTEGER, REAL, TEXT, BLOB). Composite types (struct, list, map) are not supported in the WAL and will throw at log creation time.

FastDDS Provider

A concrete PubSub implementation using eProsima Fast DDS for reliable, real-time publish/subscribe over RTPS. The provider depends only on nanoarrow (not Arrow C++), making it suitable for edge deployment.

auto provider = std::make_shared<fletcher::FastDDSPubSubProvider>(
    /*domain_id=*/0,
    /*max_payload_bytes=*/1024 * 1024  // 1 MB
);

QoS configuration

PolicySettingEffect
ReliabilityRELIABLEMiddleware retransmits lost samples
HistoryKEEP_ALLAll samples retained until acknowledged
DurabilityTRANSIENT_LOCALLate-joining subscribers receive buffered history

The provider serializes each row + attachments into a single DDS sample using the envelope wire format. On the receiver side, it deserializes and delivers raw bytes to the subscriber callback.

Zero-copy encoding

The FastDDS provider supplies a FixedWriteBuffer pointing directly at the DDS payload memory. When a publisher calls Publish(RowEncoder), the PositionalWriter writes the positional format directly into the transport buffer with no intermediate copies.

Schema delivery

For each data topic, the provider creates a companion DDS topic with the suffix /__schema using TRANSIENT_LOCAL durability and KEEP_LAST(1) QoS. The schema is serialized as Arrow IPC bytes via nanoarrow and retained for late-joining subscribers.

XRCE-DDS Provider

A PubSub implementation using eProsima Micro XRCE-DDS Client for extremely resource-constrained environments (MCUs, embedded Linux, <75 KB Flash). The XRCE-DDS standard uses a client/agent split: the thin client runs on the device while a separate Agent process bridges to the full DDS network.

fletcher::XrceConfig cfg;
cfg.transport  = fletcher::XrceTransport::kUdp;
cfg.agent_ip   = "192.168.1.10";
cfg.agent_port = 2018;
cfg.max_payload = 512;

auto provider = std::make_shared<fletcher::XrceDDSPubSubProvider>(cfg);

Schema delivery

Uses the same companion /__schema topic pattern as FastDDS with RELIABLE + TRANSIENT_LOCAL + KEEP_LAST(1) QoS. The envelope wire format is identical, so XRCE-DDS and FastDDS clients interoperate through a shared Agent.

Dependencies

Fetched automatically via CMake FetchContent (Micro-CDR v2.0.1 + Micro XRCE-DDS Client v2.4.3). No Conan package or system install required. Links PUBLIC pubsub (nanoarrow) and PRIVATE microxrcedds_client — no Arrow C++ dependency.

C API & Rust Bindings (Removed)

The codec C API (arrow_row_codec_capi.h) and its Rust bindings (Codec/bindings/rust/) have been removed. They wrapped the legacy RowCodec tagged format which has been superseded by the positional codec and generated code. The Batcher Rust bindings (Batcher/bindings/rust/) remain.

Edge binary (e.g., sensor gateway): links core + pubsub + a transport provider (fast_dds_pubsub_provider for full DDS networks, or xrce_dds_pubsub_provider for resource-constrained environments) + generated .fletcher.pb.h code. Total added dependency: nanoarrow (~100 KB). No Arrow C++ required. The web_gateway also falls on this side — it is nanoarrow-only.

Server binary (e.g., data pipeline): additionally links pubsub_arrow + codec (ArrowBridge) + row_batcher and includes .fletcher.arrow.pb.h view classes.

Building

Prerequisites

  • CMake 3.15+
  • Conan 2 package manager
  • A C++20 compiler (MSVC 2022, GCC 12+, Clang 15+)

Steps

# 1. Install dependencies via Conan
conan install . --build=missing --output-folder=build

# 2. Configure with CMake (uses the Conan-generated toolchain)
cmake --preset conan-default

# 3. Build
cmake --build build --config Release

WebClient (TypeScript)

# Install dependencies and run tests
cd WebClient
npm install
npm test            # 37 Vitest tests

# TypeScript compilation
npm run build:ts

# Generate TypeScript from .proto files
protoc --plugin=protoc-gen-fletcher=build/ProtoPlugin/Release/protoc-gen-fletcher \
       --fletcher_out=. --fletcher_opt=ts \
       -I proto your_messages.proto
Windows / Visual Studio: The root CMakeLists.txt forces /MD (MultiThreadedDLL) for all targets and patches Conan's imported targets so that Release-only packages (protobuf, Fast DDS) work correctly in Debug builds. Multi-config generators are fully supported.
