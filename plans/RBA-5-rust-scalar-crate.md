## 1. Goal & scope

Fill the existing `--fletcher_opt=rust` output (today a bare RBA-1 banner from
`EmitRustAccessor` in `protoc/src/recordbatch_accessor_emitter.cpp`) with the first
real arrow-rs accessor implementation, **for scalar columns only**, and stand up a
standalone Cargo integration crate that builds the plugin's Rust output and proves it
against arrow-rs.

In scope:

- Emit a real `<Class>Accessor` Rust struct per generated message, covering **scalar
  columns** (the `FieldKind::SCALAR` set): one `try_new(RecordBatch)` factory and one
  `from_struct(&StructArray)` factory sharing a single `from_columns` validator;
  column-oriented cast-once caching; a positional `DataType`-only type gate plus the
  `null_count()==0` enforcement for proto-non-nullable fields; typed scalar getters;
  `num_rows()`; and a `Result<Self, ArrowError>` that **never panics**.
- A new, complete, buildable crate `integration-tests/protoc-gen-fletcher-rust/`
  (`Cargo.toml`, `build.rs`, `src/lib.rs`, `tests/`, fixture `.proto`s) that runs the
  plugin with `--fletcher_opt=rust`, assembles the generated `.fletcher.rs` files into a
  `fletcher_gen` module tree, and `cargo test`s the result.
- A new CI job that installs the pinned Rust toolchain, builds the plugin, then runs
  `cargo test` on the crate.

Out of scope (composites land in **RBA-6**, which owns full Rust type parity):

- `STRUCT`, `REPEATED_SCALAR`, `REPEATED_STRUCT`, `MAP`, `NESTED_LIST` getters/storage.
  A message that contains a composite field **fails fast** (see §2 "Composite
  fail-fast"); we do not emit a half-built accessor.
- Generic metadata getters (`schema_metadata()` / `field_metadata(i)`) — RBA-6 (D-RBA-5).
  The struct **stores** `Fields` + the schema metadata map so RBA-6 adds the getters
  without reshaping the struct, but RBA-5 emits no metadata getter.
- `LargeUtf8` / `LargeBinary` / `large_list` / fixed-size / union / extension types — the
  proto→Arrow mapping never emits them (oracle §10); RBA-5 emits **no** type, getter, or
  import for them.

Pure add-on (**D-RBA-1**). The `rust` token stays opt-gated and additive: with the flag
absent nothing new is written, and **no existing generated output's bytes change** with
or without the flag. RBA-5 edits only `EmitRustAccessor` (and its private helpers) — no
existing emitter function changes behaviour. The **RBA-1 golden no-drift test stays
green**; this design adds nothing that would perturb `.fletcher.pb.h` /
`.fletcher.arrow.pb.h` / `.fletcher.ts` / `.<Message>.ipc`. Shared helpers
(`GatherFields`, `FieldInfo`, `type_mapper`) are reused **read-only**.

## 2. Rust accessor emitter shape (oracle §3/§4/§7/§8)

### 2.1 Entry point and wiring (already present)

The Rust file is already wired end-to-end by RBA-1 and must **not** be re-plumbed:

- `protoc/src/generator.cpp` parses the `rust` token (≈L2945) into `emit_rust` and, when
  set, calls `EmitRustAccessor(file)` writing `RustAccessorOutputFilename(file->name())`
  → `<stem>.fletcher.rs` (≈L3017–3023). Emission is unconditional per token and never
  content-gated (D-RBA-2).
- `protoc/src/recordbatch_accessor_emitter.{hpp,cpp}` declares/defines `EmitRustAccessor`.
  RBA-5 replaces the stub body (currently L974–985) with the real per-message emission,
  reusing the file-level `GatherFields` / `FieldInfo` / `LookupScalarArray` machinery the
  C++ path already uses.

RBA-5 emits **bare items, no `mod <pkg>` wrapper** (D-RBA-10 — the package module tree is
the crate assembler's job, §3.4). The banner already states this; keep it.

### 2.2 Per-message generated shape

For each generated message (the existing `OrderedMessages` set, `<Class>` = `ClassName(msg)`,
nested `Outer.Inner` → `Outer_Inner`) emit a `pub struct <Class>Accessor` plus its `impl`.
The shape mirrors the plan illustration (RBA-recordbatch-accessor.md L144–231) **with the
illustration's `unreachable!()` replaced by an `Err`** (D-RBA-4 — never panics). A
representative flat message `Telemetry { double temperature; int32 id; optional string label; }`:

```rust
// Generated by fletcher-protoc. DO NOT EDIT.
// Source: telemetry.proto
// Bare items — no `mod <pkg>` wrapper (assembled by build.rs, D-RBA-10).
use std::collections::HashMap;
use std::sync::Arc;
use arrow::array::{Array, ArrayRef, Float64Array, Int32Array, StringArray, StructArray};
use arrow::datatypes::{DataType, Fields};
use arrow::error::ArrowError;
use arrow::record_batch::RecordBatch;

/// Down-cast `col` to `T`, gating on Arrow type only (not name/nullable). Uses
/// arrow-rs's offset-preserving, buffer-sharing `downcast_array` (NOT a re-Arc'd
/// `downcast_ref` clone), so the cached handle keeps the column's offset/len.
fn downcast_array<T: From<arrow::array::ArrayData> + Array + 'static>(
    col: &ArrayRef, ctx: &str, expected: &DataType,
) -> Result<Arc<T>, ArrowError> {
    if col.data_type() != expected {
        return Err(ArrowError::SchemaError(format!(
            "{ctx}: expected {expected:?}, got {:?}", col.data_type())));
    }
    Ok(Arc::new(arrow::array::downcast_array::<T>(col.as_ref())))   // type checked above
}

pub struct TelemetryAccessor {
    num_rows: usize,
    fields: Fields,                       // retained for RBA-6 *_metadata()
    metadata: HashMap<String, String>,    // retained for RBA-6 *_metadata()
    temperature: Arc<Float64Array>,
    id: Arc<Int32Array>,
    label: Arc<StringArray>,
}

impl TelemetryAccessor {
    pub fn try_new(batch: RecordBatch) -> Result<Self, ArrowError> {
        let s = batch.schema();
        Self::from_columns(batch.num_rows(), batch.columns().to_vec(),
                           s.fields().clone(), s.metadata().clone())
    }

    pub fn from_struct(s: &StructArray) -> Result<Self, ArrowError> {
        // Type-test the source instead of `unreachable!()` — never panic (D-RBA-4).
        let DataType::Struct(fields) = s.data_type().clone() else {
            return Err(ArrowError::SchemaError(
                "Telemetry::from_struct: source is not a Struct".into()));
        };
        // Slice each child to the struct's [offset, offset+len) window — arrow-rs
        // StructArray::columns() are NOT pre-rebased by the struct's logical offset,
        // so a sliced struct would otherwise misalign. (D-RBA-7; carry-forward §3.)
        let cols = s.columns().iter().map(|c| c.slice(s.offset(), s.len())).collect();
        // A StructArray carries no schema-level metadata (oracle §5).
        Self::from_columns(s.len(), cols, fields, HashMap::new())
    }

    /// Shared by both factories: column-count + per-column type gate, then cache.
    fn from_columns(num_rows: usize, cols: Vec<ArrayRef>, fields: Fields,
                    metadata: HashMap<String, String>) -> Result<Self, ArrowError> {
        if cols.len() != 3 {
            return Err(ArrowError::SchemaError(format!(
                "Telemetry: expected 3 columns, got {}", cols.len())));
        }
        // proto-non-nullable fields must hold no actual nulls (oracle §4); `label`
        // is proto-optional, so it is NOT null-checked.
        for (i, name) in [(0usize, "temperature"), (1, "id")] {
            if cols[i].null_count() != 0 {
                return Err(ArrowError::SchemaError(format!(
                    "Telemetry column {i} '{name}': non-nullable, found nulls")));
            }
        }
        let temperature = downcast_array::<Float64Array>(&cols[0], "Telemetry.temperature", &DataType::Float64)?;
        let id          = downcast_array::<Int32Array>(&cols[1], "Telemetry.id", &DataType::Int32)?;
        let label       = downcast_array::<StringArray>(&cols[2], "Telemetry.label", &DataType::Utf8)?;
        Ok(Self { num_rows, fields, metadata, temperature, id, label })
    }

    pub fn num_rows(&self) -> usize { self.num_rows }

    // Non-nullable scalar getters index the cached typed column directly.
    pub fn temperature(&self, row: usize) -> f64 { self.temperature.value(row) }
    pub fn id(&self, row: usize) -> i32          { self.id.value(row) }

    // Nullable scalar → Option (utf8 → Option<&str>).
    pub fn label(&self, row: usize) -> Option<&str> {
        if self.label.is_null(row) { None } else { Some(self.label.value(row)) }
    }
}
```

### 2.3 Construction & validation rules (D-RBA-3, D-RBA-4, D-RBA-7)

- **Cast-once / column-oriented (D-RBA-3).** Each column is down-cast exactly once in
  `from_columns` and cached as `Arc<T>`; getters index the cached handle. No per-cell
  `GetScalar` / re-cast in any getter.
- **Offset-preserving cache (D-RBA-3 / D-RBA-7).** Cache via
  `arrow::array::downcast_array::<T>(col.as_ref())` wrapped once in `Arc::new(...)` — this
  shares buffers **and keeps the column's offset/len**. We do **not** use a re-`Arc`'d
  `downcast_ref` clone (`Arc::new(col.as_any().downcast_ref::<T>().unwrap().clone())`),
  which would re-wrap and is the explicitly-rejected idiom. (`downcast_ref` may still be
  used in RBA-6 only to *test* a column is a `StructArray` before recursing — never to
  cache a scalar column.)
- **Type-only positional gate (D-RBA-4).** Per column: `col.data_type() != expected`
  → `Err(ArrowError::SchemaError(..))`. `expected` is the exact `DataType` the cast
  targets (e.g. `DataType::Float64`, or `DataType::Timestamp(unit, tz)` verbatim from the
  mapper so the unit stays precise). Field **name** and the schema **nullable flag** are
  never gating — name/nullable-tolerant.
- **Column count gate (D-RBA-4).** `cols.len() != <field count>` → `Err` before any cast.
- **Actual non-nullability (D-RBA-4).** For every field the **proto** declares
  non-nullable, require `cols[i].null_count() == 0`; otherwise `Err`. Proto-optional
  (nullable) fields are **not** null-checked and surface nulls via `Option`. This is what
  makes a non-nullable getter return a plain value provably null-free.
- **Never panics (D-RBA-4).** Every failure path is an `Err`. The illustration's
  `let DataType::Struct(..) else { unreachable!() }` is replaced by an `else { return
  Err(..) }`. `from_columns` casts a column only **after** its type is checked.
- **Self-owning (D-RBA-7).** Cached `Arc<T>` own their buffers, so the accessor keeps no
  reference to the source `RecordBatch`/`StructArray` and is identical whichever factory
  built it. RBA-5 has no struct children, so it keeps **no** struct null bitmap (that is
  an RBA-6 concern); the only retained source-derived state is `fields` + `metadata` for
  RBA-6's future metadata getters.

### 2.4 from_struct child slicing — cross-language carry-forward (prominent)

**arrow-rs `StructArray::columns()` is NOT pre-windowed** by the struct's logical offset.
This is the **opposite** of C++ `arrow::StructArray::field(i)`, which already returns a
child windowed to the struct's `[offset, offset+len)`. Therefore the Rust `from_struct`
**must** slice each child explicitly:

```rust
let cols = s.columns().iter().map(|c| c.slice(s.offset(), s.len())).collect();
```

Skipping this would misalign every getter for any non-zero-offset (sliced) struct. State
this in the generated `from_struct` as a comment (as above). This is a deliberate
cross-language difference carried forward from the C++ accessor design (D-RBA-7) and is
the basis for nested composition in RBA-6 (the inner accessor is built via `from_struct`).

### 2.5 Scalar arrow-rs types covered

Mirror the C++ `LookupScalarArray` table (`recordbatch_accessor_emitter.cpp` L58–95), one
arrow-rs concrete array + getter return type per scalar mapper entry:

| Proto / mapper `arrow_type_expr` | `expected` `DataType` | arrow-rs array (cached `Arc<T>`) | Getter return (non-null / null) |
|---|---|---|---|
| `boolean()` | `Boolean` | `BooleanArray` | `bool` / `Option<bool>` |
| `int8()` | `Int8` | `Int8Array` | `i8` / `Option<i8>` |
| `int16()` | `Int16` | `Int16Array` | `i16` / `Option<i16>` |
| `int32()` | `Int32` | `Int32Array` | `i32` / `Option<i32>` |
| `int64()` | `Int64` | `Int64Array` | `i64` / `Option<i64>` |
| `uint8()` | `UInt8` | `UInt8Array` | `u8` / `Option<u8>` |
| `uint16()` | `UInt16` | `UInt16Array` | `u16` / `Option<u16>` |
| `uint32()` | `UInt32` | `UInt32Array` | `u32` / `Option<u32>` |
| `uint64()` | `UInt64` | `UInt64Array` | `u64` / `Option<u64>` |
| `float32()` | `Float32` | `Float32Array` | `f32` / `Option<f32>` |
| `float64()` | `Float64` | `Float64Array` | `f64` / `Option<f64>` |
| `utf8()` | `Utf8` | `StringArray` | `&str` / `Option<&str>` |
| `binary()` | `Binary` | `BinaryArray` | `&[u8]` / `Option<&[u8]>` |
| `timestamp(unit[,tz])` | `Timestamp(unit, tz)` | `Timestamp<Unit>Array` | `i64` / `Option<i64>` |
| `duration(unit)` | `Duration(unit)` | `Duration<Unit>Array` | `i64` / `Option<i64>` |

Notes:

- **utf8 → `&str`, binary → `&[u8]`** (established decision 1; mirrors C++ borrowed
  byte-view, D-RBA-8). `&str` requires valid UTF-8, so binary **must** be `&[u8]` — there
  is **no "binary-as-`&str`"** anywhere. Nullable variants are `Option<&str>` /
  `Option<&[u8]>`. The borrowed return ties the getter to `&self`, which is correct: the
  cached `Arc<StringArray>`/`Arc<BinaryArray>` keeps the buffer alive for the borrow.
- **No `LargeUtf8`/`LargeBinary`** types, getters, or imports (established decision 2;
  oracle §10). The mapper never produces them; if the gate ever sees `LargeUtf8`/`LargeBinary`
  at runtime the type-equality check simply errors (`expected Utf8, got LargeUtf8`), which
  is the desired "unexpected layout is a validation error" behaviour (oracle §10).
- **Timestamp/duration unit.** As in C++, the cached array class is unit-generic but
  `expected` carries the exact unit (and timestamp timezone) verbatim from the mapper's
  `arrow_type_expr`, so the gate stays unit-precise. The concrete arrow-rs array name
  (`TimestampNanosecondArray`, `DurationMillisecondArray`, …) is selected per unit at emit
  time; getter returns the raw `i64` (matching the C++ `int64_t` choice).
- Only `is_supported_scalar` / `FieldKind::SCALAR` fields are emitted in RBA-5.

### 2.6 Composite fail-fast

A message whose `GatherFields` yields **any** non-`SCALAR` `FieldKind`
(`STRUCT`/`REPEATED_SCALAR`/`REPEATED_STRUCT`/`MAP`/`NESTED_LIST`) is not yet supported by
the Rust emitter. RBA-5 fails fast rather than emitting a half-accessor. Preferred
mechanism (confirm at implementation):

- **Generation-time:** emit a clear `// fletcher: <Class> has composite field
  '<name>' (<kind>) — Rust accessor support lands in RBA-6` comment **in place of** the
  struct (no partial struct, no partial getters), exactly as the existing path emits
  generation-time comments for unsupported constructs (oracle §6, "no silent gap"). The
  RBA-5 crate's fixtures are scalar-only, so no fixture trips this; the comment documents
  the boundary for any consumer who points the `rust` opt at a composite message.

(The C++ emitter already supports composites through RBA-4; the Rust gap is RBA-5-specific
and closes in RBA-6. The fail-fast must never produce code that fails to compile.)

## 3. The new Cargo crate `integration-tests/protoc-gen-fletcher-rust/`

The repo has **no Rust today**, so every piece is new and spelled out here. The crate
mirrors the existing `integration-tests/protoc-gen-fletcher-npm/` consumer harness: it
consumes the **locally built** plugin (never a published/downloaded artifact), generates
from fixture `.proto`s at build time, and asserts behaviour in tests.

Layout:

```
integration-tests/protoc-gen-fletcher-rust/
  Cargo.toml
  build.rs
  README.md
  proto/
    telemetry.proto            # pkg fletcher.rba.telem  — flat scalar fixture
    sensors_a.proto            # pkg fletcher.rba.shared  — same-package file A
    sensors_b.proto            # pkg fletcher.rba.shared  — same-package file B
  src/
    lib.rs                     # include!s the assembled fletcher_gen.rs
  tests/
    scalar_columns.rs          # forcing test + acceptance tests
```

### 3.1 `Cargo.toml`

Standalone crate (its own `[package]`; **not** a workspace member — the repo has no
workspace and RBA-5 adds none). Exact arrow pin (established decision 4 — but verify the
version is real, §8):

```toml
[package]
name = "protoc-gen-fletcher-rust"
version = "0.0.0"
edition = "2021"
publish = false
license = "LGPL-3.0-or-later"

[lib]
path = "src/lib.rs"

[dependencies]
# Exact pin: the generated code commits to this arrow-rs API surface
# (downcast_array, From<ArrayData>, Fields, StructArray). MUST verify 59.0.0 is a
# real released version whose API matches before merge — see §8.
arrow = "=59.0.0"

[dev-dependencies]
# arrow is also used in tests to build RecordBatches/StructArrays; the [dependencies]
# entry above is in scope for tests, so no separate dev-dep is required. Add others
# (e.g. an assertion helper) only if a test needs them — keep deps minimal & offline.
```

No network at build or test time other than the one-time `cargo` registry fetch of the
pinned `arrow` crate (which CI caches/vendors per §5). `build.rs` itself never downloads.

### 3.2 `build.rs` — job 1: run the plugin

`build.rs` locates `protoc` and the **built** `protoc-gen-fletcher` plugin, then runs the
plugin on every fixture `.proto` with `--fletcher_opt=rust`, emitting `<stem>.fletcher.rs`
into `OUT_DIR`. Contract:

1. Read `PROTOC` (path to `protoc`) and `FLETCHER_PROTOC_PLUGIN` (absolute path to the
   built plugin binary) from the environment (§4). If either is unset or the path does not
   exist, **fail the build with a clear message** (e.g. `error: FLETCHER_PROTOC_PLUGIN is
   not set or does not point at a fletcher-protoc binary; build the plugin first (see
   README)`). No fallback download.
2. For each fixture, invoke:
   ```
   $PROTOC --plugin=protoc-gen-fletcher=$FLETCHER_PROTOC_PLUGIN \
           --fletcher_opt=rust --fletcher_out=$OUT_DIR \
           -I proto proto/<file>.proto
   ```
   protoc derives the plugin executable from the `--plugin=protoc-gen-fletcher=<path>`
   mapping (the same `protoc-gen-<name>` convention the npm shim relies on). On Windows
   the `.exe` suffix is part of the path the env var carries.
3. Surface the plugin's stderr on failure and return a non-zero build result.
4. `println!("cargo:rerun-if-changed=proto/...")` for each fixture and
   `cargo:rerun-if-env-changed=FLETCHER_PROTOC_PLUGIN` / `PROTOC` so a plugin rebuild or a
   fixture edit re-triggers generation.

`OUT_DIR` is the standard Cargo-provided per-build output directory; nothing is written
into the source tree.

### 3.3 `build.rs` — job 2: the module assembler (D-RBA-10)

The generated `.fletcher.rs` files carry **bare accessor items with no `mod <pkg>`
wrapper** (§2.1). The assembler (this second `build.rs` job) declares the `fletcher_gen`
module tree **exactly once** and `include!`s every generated file into the right module —
prost's group-by-package model realised with per-file `include!`s. Algorithm:

1. After generation, enumerate the `*.fletcher.rs` files in `OUT_DIR`.
2. Determine each file's **proto package** (key on package, **not** file stem — stems may
   contain `-`/`.`/path separators and collide across directories, D-RBA-10). The package
   is known to `build.rs` from the fixture set (the simplest robust approach: a small
   static map of fixture file → package maintained alongside the fixtures, or parse the
   `package` line). Confirm the discovery mechanism at implementation; the fixtures' pinned
   `package` lines (§6) make this deterministic.
3. Group files by package and emit a single aggregator `fletcher_gen.rs` into `OUT_DIR`
   that:
   - declares `pub mod fletcher_gen { ... }` once;
   - for each package `a.b.c`, declares the nested `pub mod a { pub mod b { pub mod c { ...
     } } }` chain **once** (a package module is declared exactly once no matter how many
     files contribute);
   - inside a package's innermost module, `include!("<abs path to each .fletcher.rs of
     that package>");` — so two files in the **same** package both mount into the **one**
     `mod`, never duplicating it;
   - a no-package file's items `include!` directly under `fletcher_gen`.
   - Segment sanitization (single rule, D-RBA-10 / oracle §8.1): a segment that is a Rust
     keyword → `r#<seg>`; otherwise proto segments are valid idents (no rename); a segment
     invalid even as `r#…` is a generation/assembly error.

   Sketch of the emitted aggregator for the §6 fixtures:
   ```rust
   pub mod fletcher_gen {
       pub mod fletcher {
           pub mod rba {
               pub mod telem {
                   include!(concat!(env!("OUT_DIR"), "/telemetry.fletcher.rs"));
               }
               pub mod shared {                       // ONE mod for both files
                   include!(concat!(env!("OUT_DIR"), "/sensors_a.fletcher.rs"));
                   include!(concat!(env!("OUT_DIR"), "/sensors_b.fletcher.rs"));
               }
           }
       }
   }
   ```
   This yields the D-RBA-10 path `crate::fletcher_gen::fletcher::rba::telem::TelemetryAccessor`
   etc., mirroring the C++ `fletcher_gen::<pkg>` 1:1.
4. `src/lib.rs` mounts the aggregator with a single `include!`:
   ```rust
   include!(concat!(env!("OUT_DIR"), "/fletcher_gen.rs"));
   ```

The `fletcher_gen` mount-point name and the package-path scheme are locked (D-RBA-10 —
changing them after RBA-5 lands is STOP-AND-ASK).

### 3.4 `src/lib.rs` and `tests/`

`src/lib.rs` is the single `include!` above (plus the crate license header). `tests/`
holds `cargo test` cases (§6) that reference accessors by their fully-qualified
`crate::fletcher_gen::<pkg>::<Class>Accessor` paths — which is why the fixtures have pinned
`package` lines (established decision 6).

## 4. Plugin-discovery contract

`build.rs` finds the tools through **two environment variables** — no network, no
download (unlike the npm shim's GitHub-Release path):

| Env var | Meaning | Set by |
|---|---|---|
| `PROTOC` | Absolute path to the `protoc` executable. | The CI step / local dev (devcontainer ships protoc; Conan can also surface it). Falls back to `protoc` on `PATH` only if explicitly chosen at implementation; preferred is an explicit path. |
| `FLETCHER_PROTOC_PLUGIN` | Absolute path to the freshly built `protoc-gen-fletcher` (`fletcher-protoc`) binary (`.exe` on Windows). | The CI step / local dev, after `conan create protoc/.`, by locating the binary in the Conan cache (the same `find ~/.conan2/p -path '*/p/bin/fletcher-protoc' …` pattern the npm CI uses, L59–60). |

How the C++/Conan build exports the plugin path: the existing pattern (npm CI, L57–64)
builds the plugin with `conan create protoc/. --build=missing -pr:a=$PROFILE`, then
locates the binary in the Conan cache via `find`. RBA-5 reuses exactly that locate step
and assigns the result to `FLETCHER_PROTOC_PLUGIN` (instead of priming an npm cache dir).
No published artifact, no registry, no download.

`build.rs` **fails clearly** if either var is unset or its path is missing (§3.2 item 1),
satisfying the acceptance "`build.rs` fails clearly if the plugin/protoc is absent".

## 5. CI wiring

**Must-confirm filename (§8).** The existing per-consumer integration workflows live under
`.github/workflows/` and are named `ci.integration-test.<name>.yml` (confirmed: e.g.
`ci.integration-test.protoc-gen-fletcher-npm.yml`, `ci.integration-test.protoc-arrow-bridge.yml`).
RBA-5 adds **`.github/workflows/ci.integration-test.protoc-gen-fletcher-rust.yml`**,
mirroring the npm workflow's structure, and registers it in the aggregator
**`.github/workflows/ci.pr.yml`** as a new `workflow_call` job (the npm workflow is invoked
from `ci.pr.yml`; the Rust one must be wired in the same place). **Confirm both filenames
at implementation** — `ci.pr.yml` is the PR aggregator; the exact `jobs:`/`uses:` wiring
must match how `ci.integration-test.protoc-gen-fletcher-npm.yml` is referenced.

New job/step sequence (Linux in the devcontainer image, mirroring npm's Linux job;

### 6.1 Fixtures (pinned `package` lines)

Pinned `package` lines make the `crate::fletcher_gen::<pkg>::<Class>Accessor` paths
testable (established decision 6) and exercise D-RBA-10's same-package multi-file case:

### 6.2 Forcing test — `scalar_columns_read_and_validate_positionally`

One red-first forcing test in `tests/scalar_columns.rs` (oracle §9: one forcing test per
item; remaining are green acceptance tests added in the same item):

1. **Build a `RecordBatch`** matching `Telemetry`'s schema (Float64, Int32, Utf8 with a
   null row, Binary, Binary with a null row).
2. **Construct** `TelemetryAccessor::try_new(batch.clone())` → `Ok`.
3. **Per-row scalar reads, positional:** assert `acc.temperature(r)`, `acc.id(r)`,
   `acc.blob(r)` (as `&[u8]`) for each row.
4. **Nullable `Option`:** assert `acc.label(null_row) == None` and `Some("…")` on a
   populated row; same for `acc.opt_blob` (`None` / `Some(&[u8])`).
5. **Type-mismatch → `Err`:** build a batch where one column has the wrong Arrow type
   (e.g. `Int64` where `Int32` is expected) and assert `try_new` returns `Err` (never
   panics).
6. **Name-mismatch, type-compatible → `Ok`:** build a batch with the **same column types**
   but **different field names** (and/or different nullable flags) and assert `try_new`
   returns `Ok` and reads the same values — proving §4 name/flag tolerance.
7. **`from_struct` parity:** wrap the **same** columns as a `StructArray` (including a
   non-zero offset via `.slice(off, len)` to exercise §2.4 child-slicing) and assert
   `TelemetryAccessor::from_struct(&sa)` reads **identically** to the `RecordBatch` path.
8. **Wrong column count → `Err`** (acceptance): a batch with too few/many columns →
   `Err`, never panic.
9. **Non-nullable carries a runtime null → `Err`** (acceptance, the §4 `null_count`
   gate): a batch whose `temperature`/`id`/`blob` column actually contains a null → `Err`.

Same-package mounting is exercised by a small acceptance test that constructs
`crate::fletcher_gen::fletcher::rba::shared::PressureAccessor` **and**
`…::shared::HumidityAccessor` from minimal batches — proving both same-package files mounted
into the one module (D-RBA-10) and the path scheme resolves.

Acceptance (from the plan, L902–904): wrong column count → `Err`; `try_new` never panics;
the crate **builds and tests green in CI** with the pinned toolchain + pinned `arrow`;

## 9. Locked-decisions conformance

- **D-RBA-1 (pure add-on).** Only `EmitRustAccessor` + new crate/CI files; no existing
  output bytes change, no existing emitter behaviour changes; shared helpers reused
  read-only; RBA-1 no-drift test stays green. The `rust` token stays opt-gated/additive.
- **D-RBA-3 (column-oriented, cast-once).** One down-cast per column at construction,
  cached as `Arc<T>`; getters index the cache; **no** per-cell `GetScalar`/re-cast; the
  cache uses the offset-preserving `arrow::array::downcast_array::<T>` idiom, **not** a
  re-`Arc`'d `downcast_ref` clone.
- **D-RBA-4 (positional type-check → `Result`).** Column-count + per-column type-only gate
  (`DataType` equality; name/nullable-flag tolerant); proto-non-nullable fields enforce
  `null_count()==0`; returns `Result<Self, ArrowError>` and **never panics** (the
  illustration's `unreachable!()` is replaced by `Err`); a column is cast only after its
  type is checked.
- **D-RBA-8 (C++/Rust parity from one model).** Rust is emitted from the same
  `FieldInfo`/`type_mapper` model; the scalar type set mirrors C++ `LookupScalarArray`;
  utf8→`&str`, binary→`&[u8]` mirror the C++ borrowed byte-view.
- **D-RBA-10 (cross-file module convention).** Generated `.fletcher.rs` carry bare items
  with no `mod <pkg>` wrapper; the `build.rs` assembler declares the `fletcher_gen` tree
  once, groups files by proto package, and `include!`s every file of a package into that
  package's single `mod` — same-package multi-file mounting never duplicates a `mod`;
  `src/lib.rs` `include!`s the aggregator; segment sanitization follows the single
  keyword→`r#` rule. Mount-point name and path scheme are locked.
- (Carried, RBA-6.) **D-RBA-7** struct child-slicing in `from_struct` is implemented now
  (§2.4) as the basis for RBA-6 nested composition, even though RBA-5 has no struct
  children; the retained `Fields`/`metadata` reserve room for RBA-6's metadata getters
  without reshaping the struct.

## Review fixes (RBA-5, post-review)

5. **build.rs discovery — env vars authoritative (§4).** `PROTOC` /
   `FLETCHER_PROTOC_PLUGIN` are authoritative: if set they are used (a missing
